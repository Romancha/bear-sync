# bear-sync: Архитектурный дизайн

## Проблема

ИИ-ассистент (openclaw) работает на VPS с Ubuntu 24. Основное хранилище заметок — Bear (macOS/iOS). Нужно обеспечить openclaw доступ к заметкам: чтение 24/7, создание/редактирование с очередью когда Mac офлайн.

## Контекст

- **Bear** — только macOS/iOS, нет HTTP API. Есть x-callback-url API (запись) и SQLite база (чтение).
- **openclaw** — Telegram-бот на VPS, может выполнять Python-скрипты.
- **Mac** — MacBook, доступен в рабочее время (не 24/7).
- **Заметки** — 500-2000, с вложениями (картинки, PDF). Текущее состояние: 511 заметок, 108 тегов, 428 вложений (71 файл + 357 изображений), 101 бэклинк.
- **Задержка** — минуты допустимы.

## Рассмотренные альтернативы

| Подход | Плюсы | Минусы | Вердикт |
|---|---|---|---|
| Obsidian + iCloud + Git | Бесплатно, Git-нативно, openclaw работает с .md | Obsidian не понравился по UX | Отклонён |
| Self-hosted (Joplin/Memos/SiYuan/Trilium/Anytype) | Заметки на VPS, прямой доступ | Ни один не совмещает хороший Apple UI + серверный API | Отклонён |
| Облачные с API (Notion) | API всегда доступен | Vendor lock-in, данные не у тебя | Отклонён |
| **Bear + hub + очередь** | Bear остаётся UI, чтение 24/7, запись через очередь | Нужен Mac-агент + hub-сервис | **Выбран** |

## Архитектура: Hub-модель

```
Bear (Mac/iPhone)                    openclaw (VPS)
     |  iCloud                            |
Mac-агент ──push/pull──>  bear-sync API  <──read/write── openclaw
(читает Bear SQLite,     (центральный      (Telegram-бот,
 пишет через x-callback)  хаб заметок)      RAG, поиск)
```

### Принципы

1. **bear-sync API** — центральный хаб, к которому обращаются и Mac-агент, и openclaw как независимые клиенты.
2. **Decoupling** — Mac-агент и openclaw не знают друг о друге. Замена Bear на другое приложение меняет только Mac-агент.
3. **Хаб — источник правды для openclaw.** Bear — источник правды для пользователя. Хаб — read replica + write queue.
4. **Хаб живёт на VPS** (тот же сервер, отдельный порт).

## Компоненты

### 1. bear-sync API (Go)

Центральный хаб на VPS. Go выбран за: один бинарник, ~10 MB RAM, простой деплой.

**Стек:** Go + chi (REST, net/http-совместимый роутер) + SQLite (modernc.org/sqlite, pure Go)

**SQLite конфигурация:** при инициализации БД: `PRAGMA journal_mode=WAL` + `PRAGMA busy_timeout=5000` + `PRAGMA foreign_keys=ON`. WAL нужен для конкурентного доступа (Mac-агент пишет, openclaw читает одновременно). `foreign_keys=ON` обязателен — SQLite по умолчанию не enforce FK constraints.

### Схема БД хаба

Подход: **чистая схема + raw JSON + dual-id** (Вариант C+). Своя схема с читаемыми именами. **Dual-id:** `id` — hub-generated UUID (стабильный PK), `bear_id` — UUID из Bear (UNIQUE, NULL до первого sync). Каждая таблица хранит `bear_raw TEXT` — JSON-снэпшот оригинальной строки Bear для страховки от потери данных. FK с `ON DELETE CASCADE`.

#### Рассмотренные варианты схемы

| Вариант | Описание | Итог |
|---|---|---|
| A: Зеркало 1:1 | Те же таблицы и колонки как в Bear | Простой Mac-агент, но проблема Z_PK в junction tables, нечитаемый API |
| B: Чистая схема | Свои имена, UUID как PK, ISO даты | Чистый API, но нет страховки от потери полей при маппинге |
| **C: Чистая + raw JSON** | Как B, но с JSON-дампом оригинала | Чистый API + полная целостность данных. ~15 MB лишнего места — незначительно |
| D: Двойная БД | bear_mirror.sqlite + hub.sqlite | Максимальная гибкость, но over-engineering: три компонента вместо двух |

```sql
-- ============================================
-- NOTES (зеркало ZSFNOTE, 38 колонок → чистые имена)
-- ============================================
-- Dual-id: id — hub-generated UUID (стабильный PK), bear_id — UUID из Bear (заполняется после sync).
-- При создании через openclaw (Mac офлайн): id генерируется хабом, bear_id=NULL.
-- После apply в Bear: bridge присылает bear_id через ack, хаб обновляет.
CREATE TABLE notes (
    rowid               INTEGER PRIMARY KEY AUTOINCREMENT,  -- stable rowid для FTS5 content-sync
    id                  TEXT NOT NULL UNIQUE,                -- Hub UUID (generated by hub, hex(randomblob(16)))
    bear_id             TEXT UNIQUE,        -- Bear ZUNIQUEIDENTIFIER (NULL до первого sync с Bear)
    title               TEXT NOT NULL,
    subtitle            TEXT,
    body                TEXT NOT NULL,       -- Markdown (Bear ZTEXT)

    -- Флаги
    archived            INTEGER DEFAULT 0,
    encrypted           INTEGER DEFAULT 0,
    has_files           INTEGER DEFAULT 0,
    has_images          INTEGER DEFAULT 0,
    has_source_code     INTEGER DEFAULT 0,
    locked              INTEGER DEFAULT 0,
    pinned              INTEGER DEFAULT 0,
    shown_in_today      INTEGER DEFAULT 0,
    trashed             INTEGER DEFAULT 0,
    permanently_deleted INTEGER DEFAULT 0,
    skip_sync           INTEGER DEFAULT 0,
    todo_completed      INTEGER DEFAULT 0,
    todo_incompleted    INTEGER DEFAULT 0,
    version             INTEGER DEFAULT 0,

    -- Даты (ISO 8601, конвертированы из Core Data epoch + 978307200)
    created_at          TEXT,
    modified_at         TEXT,
    archived_at         TEXT,
    encrypted_at        TEXT,
    locked_at           TEXT,
    pinned_at           TEXT,
    trashed_at          TEXT,
    order_date          TEXT,
    conflict_id_date    TEXT,

    -- Метаданные
    last_editing_device TEXT,
    conflict_id         TEXT,               -- ZCONFLICTUNIQUEIDENTIFIER
    encryption_id       TEXT,               -- ZENCRYPTIONUNIQUEIDENTIFIER
    encrypted_data      BLOB,               -- для зашифрованных заметок

    -- Hub-only поля
    sync_status         TEXT DEFAULT 'synced',  -- synced | pending_to_bear | conflict
    hub_modified_at     TEXT,                    -- когда хаб последний раз менял запись
    bear_raw            TEXT                     -- JSON snapshot оригинальной строки Bear
);

-- ============================================
-- TAGS (зеркало ZSFNOTETAG, 22 колонки)
-- ============================================
CREATE TABLE tags (
    id                      TEXT PRIMARY KEY,   -- Hub UUID
    bear_id                 TEXT UNIQUE,        -- Bear ZUNIQUEIDENTIFIER (NULL если создан через openclaw)
    title                   TEXT NOT NULL UNIQUE, -- Имя тега (иерархия через "/": "08_Knowledge/IT"). UNIQUE — теги уникальны по имени в Bear.
    pinned                  INTEGER DEFAULT 0,
    is_root                 INTEGER DEFAULT 0,
    hide_subtags_notes      INTEGER DEFAULT 0,
    sorting                 INTEGER DEFAULT 0,
    sorting_direction       INTEGER DEFAULT 0,
    encrypted               INTEGER DEFAULT 0,
    version                 INTEGER DEFAULT 0,

    -- Даты (ISO 8601)
    modified_at             TEXT,
    pinned_at               TEXT,
    pinned_notes_at         TEXT,
    encrypted_at            TEXT,
    hide_subtags_notes_at   TEXT,
    sorting_at              TEXT,
    sorting_direction_at    TEXT,
    tag_con_date            TEXT,
    tag_con                 TEXT,

    bear_raw                TEXT
);

-- ============================================
-- NOTE_TAGS (зеркало Z_5TAGS junction table)
-- ============================================
CREATE TABLE note_tags (
    note_id TEXT NOT NULL REFERENCES notes(id) ON DELETE CASCADE,
    tag_id  TEXT NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
    PRIMARY KEY (note_id, tag_id)
);

-- ============================================
-- PINNED_NOTE_TAGS (зеркало Z_5PINNEDINTAGS)
-- ============================================
CREATE TABLE pinned_note_tags (
    note_id TEXT NOT NULL REFERENCES notes(id) ON DELETE CASCADE,
    tag_id  TEXT NOT NULL REFERENCES tags(id) ON DELETE CASCADE,
    PRIMARY KEY (note_id, tag_id)
);

-- ============================================
-- ATTACHMENTS (зеркало ZSFNOTEFILE, 35 колонок)
-- Z_ENT: 8=файл, 9=изображение, 10=видео
-- ============================================
CREATE TABLE attachments (
    id                      TEXT PRIMARY KEY,   -- Hub UUID
    bear_id                 TEXT UNIQUE,        -- Bear ZUNIQUEIDENTIFIER
    note_id                 TEXT REFERENCES notes(id) ON DELETE CASCADE,
    type                    TEXT NOT NULL,       -- 'file' | 'image' | 'video' (из Z_ENT)
    filename                TEXT,
    normalized_extension    TEXT,
    file_size               INTEGER,
    file_index              INTEGER,

    -- Медиа-метаданные
    width                   INTEGER,
    height                  INTEGER,
    animated                INTEGER DEFAULT 0,
    duration                INTEGER,            -- для видео
    width1                  INTEGER,            -- thumbnail
    height1                 INTEGER,

    -- Флаги
    downloaded              INTEGER DEFAULT 0,
    encrypted               INTEGER DEFAULT 0,
    permanently_deleted     INTEGER DEFAULT 0,
    skip_sync               INTEGER DEFAULT 0,
    unused                  INTEGER DEFAULT 0,
    uploaded                INTEGER DEFAULT 0,
    version                 INTEGER DEFAULT 0,

    -- Даты (ISO 8601)
    created_at              TEXT,
    modified_at             TEXT,
    inserted_at             TEXT,
    encrypted_at            TEXT,
    unused_at               TEXT,
    uploaded_at             TEXT,
    search_text_at          TEXT,

    -- Метаданные
    last_editing_device     TEXT,
    encryption_id           TEXT,
    search_text             TEXT,
    encrypted_data          BLOB,

    -- Файл на диске VPS
    file_path               TEXT,

    bear_raw                TEXT
);

-- ============================================
-- BACKLINKS (зеркало ZSFNOTEBACKLINK, 11 колонок)
-- ============================================
CREATE TABLE backlinks (
    id              TEXT PRIMARY KEY,           -- Hub UUID
    bear_id         TEXT UNIQUE,               -- Bear ZUNIQUEIDENTIFIER
    linked_by_id    TEXT REFERENCES notes(id) ON DELETE CASCADE,  -- заметка, которая ссылается
    linking_to_id   TEXT REFERENCES notes(id) ON DELETE CASCADE,  -- заметка, на которую ссылаются
    title           TEXT,
    location        INTEGER,
    version         INTEGER DEFAULT 0,
    modified_at     TEXT,

    bear_raw        TEXT
);

-- ============================================
-- HUB-ONLY ТАБЛИЦЫ
-- ============================================

-- Очередь записи (openclaw → Bear через Mac-агент)
-- Семантика доставки:
--   1. openclaw → hub: effectively-once через client-supplied Idempotency-Key.
--      Повторный запрос с тем же ключом возвращает существующую queue item (без дубля).
--   2. hub → bridge: at-least-once (lease-based processing, возможен повторный pickup после lease expiry).
--   3. bridge apply: "duplicate-safe" за счёт pre-apply проверки фактического состояния в Bear
--      (уже создано/обновлено/тег уже добавлен) + идемпотентного ack на хабе.
--   4. POST /api/sync/ack идемпотентен: повторный ack того же idempotency_key — no-op.
-- Bridge local state (~/.bear-bridge-state.json → applied_idempotency_keys) —
-- оптимизация (skip network call), не источник правды для дедупликации.
CREATE TABLE write_queue (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    idempotency_key TEXT UNIQUE NOT NULL,  -- всегда передаётся клиентом, без DEFAULT (чтобы не скрыть баг пропуска ключа)
    action          TEXT NOT NULL,           -- create | update | add_tag | trash (remove_tag отложен до Фазы 3)
    note_id         TEXT,                    -- hub id заметки (может быть NULL для create с inline payload)
    payload         TEXT NOT NULL,           -- JSON с данными операции
    created_at      TEXT DEFAULT (datetime('now')),
    status          TEXT DEFAULT 'pending',  -- pending | processing | applied | failed
    processing_by   TEXT,                    -- идентификатор bridge instance (hostname/pid)
    lease_until     TEXT,                    -- lease expiry (ISO 8601). NULL если не в processing.
    applied_at      TEXT,
    error           TEXT
);

-- Метаданные синхронизации (key-value)
CREATE TABLE sync_meta (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- Ключи: last_sync_at, last_push_at, queue_size, bear_db_hash

-- ============================================
-- ИНДЕКСЫ
-- ============================================
CREATE INDEX idx_notes_modified ON notes(modified_at);
CREATE INDEX idx_notes_trashed ON notes(trashed);
CREATE INDEX idx_notes_sync_status ON notes(sync_status);
CREATE INDEX idx_attachments_note ON attachments(note_id);
CREATE INDEX idx_backlinks_linked_by ON backlinks(linked_by_id);
CREATE INDEX idx_backlinks_linking_to ON backlinks(linking_to_id);
CREATE INDEX idx_write_queue_status ON write_queue(status);
CREATE INDEX idx_write_queue_lease ON write_queue(status, lease_until);  -- для lease expiry check

-- Полнотекстовый поиск по заметкам
-- content_rowid='rowid' ссылается на notes.rowid (INTEGER PRIMARY KEY AUTOINCREMENT) — стабильный, не меняется при VACUUM.
CREATE VIRTUAL TABLE notes_fts USING fts5(
    title, body, content='notes', content_rowid='rowid'
);

-- FTS5 триггеры для автоматической синхронизации индекса
CREATE TRIGGER notes_fts_insert AFTER INSERT ON notes BEGIN
    INSERT INTO notes_fts(rowid, title, body) VALUES (new.rowid, new.title, new.body);
END;

CREATE TRIGGER notes_fts_update AFTER UPDATE ON notes BEGIN
    INSERT INTO notes_fts(notes_fts, rowid, title, body) VALUES ('delete', old.rowid, old.title, old.body);
    INSERT INTO notes_fts(rowid, title, body) VALUES (new.rowid, new.title, new.body);
END;

CREATE TRIGGER notes_fts_delete AFTER DELETE ON notes BEGIN
    INSERT INTO notes_fts(notes_fts, rowid, title, body) VALUES ('delete', old.rowid, old.title, old.body);
END;
```

### Структура Bear на диске

```
~/Library/Group Containers/9K33E3U3T4.net.shinyfrog.bear/Application Data/
├── database.sqlite                    -- основная БД (Core Data store)
└── Local Files/
    ├── Note Images/<UUID>/filename    -- изображения (Z_ENT=9)
    └── Note Files/<UUID>/filename     -- файлы (Z_ENT=8)
```

Bear хранит даты в **Core Data epoch** (секунды от 2001-01-01). Конвертация в Unix: `unix = core_data + 978307200`.

### API эндпоинты

**Транспорт:** HTTPS обязателен. Caddy reverse proxy с автоматическими Let's Encrypt сертификатами. bear-sync слушает `127.0.0.1:PORT` (только localhost), Caddy проксирует внешний трафик с TLS-терминацией.

**Авторизация:** `Authorization: Bearer <token>`. Отдельные токены для Mac-агента и openclaw (разные права).

**Лимиты:**
- Максимальный размер body запроса: 50 MB для `sync/push` (initial sync батчей), 1 MB для openclaw API, 100 MB для `sync/attachments/:id`.
- Rate limiting: настраивается в Caddy (`rate_limit` directive) — 60 req/min для openclaw, 120 req/min для bridge (bridge ходит через интернет, не localhost — при утечке токена без лимита возможен DoS).
- Go middleware: `http.MaxBytesReader` для enforce body size.

```
=== API для openclaw ===
-- Все :id в API — hub UUID (стабильный). bear_id доступен в ответах как поле.
-- Для POST/PUT/DELETE обязателен заголовок: Idempotency-Key: <uuid>
-- Повтор с тем же ключом должен вернуть существующий результат (без новой queue item).

GET    /api/notes                     -- список заметок (без body — только title, subtitle, мета, теги)
  ?tag=<name>                         -- фильтр по тегу
  ?trashed=true                       -- включая корзину
  ?encrypted=false                    -- исключить зашифрованные
  ?limit=50&offset=0                  -- пагинация
  ?sort=modified_at&order=desc        -- сортировка

GET    /api/notes/:id                 -- одна заметка (body + tags + backlinks)

GET    /api/notes/search?q=...        -- FTS5 поиск по title+body
                                      -- Контракт роутинга: /search регистрируется как static route ПЕРЕД :id.
                                      -- chi приоритизирует static routes над параметризованными автоматически.
  &tag=<name>                         -- в рамках тега
  &limit=20

POST   /api/notes                     -- создать заметку → write_queue
PUT    /api/notes/:id                 -- обновить → write_queue
DELETE /api/notes/:id                 -- в корзину → write_queue

GET    /api/tags                      -- все теги (с иерархией)
POST   /api/notes/:id/tags           -- добавить тег → write_queue
-- DELETE /api/notes/:id/tags/:tag_id — ОТЛОЖЕНО до Фазы 3.
-- Bear x-callback-url не имеет эндпоинта для удаления тега.
-- Workaround через body rewrite рискует повредить inline-теги в тексте.
-- См. "Ограничения и edge cases" → remove_tag.

GET    /api/attachments/:id           -- скачать файл
GET    /api/notes/:id/backlinks       -- заметки, ссылающиеся на эту

GET    /api/sync/status               -- статус синхронизации


=== API для Mac-агента (sync) ===

POST   /api/sync/push                 -- пушит дельту
  Body: {
    "notes": [...],                   -- upsert по bear_id. Хаб ищет по bear_id → обновляет, не найдено → создаёт (hub id + bear_id).
                                      -- Если заметка имеет sync_status=pending_to_bear (есть непримененные изменения от openclaw):
                                      --   bear_raw и Bear-поля (флаги, даты) обновляются из push,
                                      --   но body/title НЕ перезаписываются — приоритет у версии openclaw.
                                      --   sync_status остаётся pending_to_bear до ack от bridge.
    "tags": [...],                    -- upsert по bear_id, аналогично
    "note_tags": [...],               -- связи только для заметок в этом push. Хаб делает:
                                      --   DELETE FROM note_tags WHERE note_id IN (<hub ids заметок из этого push>)
                                      --   затем INSERT новых привязок. Привязки заметок НЕ в push — не затрагиваются.
    "pinned_note_tags": [...],        -- аналогично note_tags, scope ограничен заметками из push
    "attachments": [...],             -- upsert по bear_id
    "backlinks": [...],               -- upsert по bear_id
    "deleted_note_ids": [...],        -- bear_id удалённых заметок. Каскадно удаляет attachments, backlinks, note_tags
    "deleted_tag_ids": [...],          -- bear_id тегов. Каскадно удаляет note_tags, pinned_note_tags (ON DELETE CASCADE)
    "deleted_attachment_ids": [...],   -- bear_id вложений
    "deleted_backlink_ids": [...]     -- bear_id бэклинков
  }

GET    /api/sync/queue                -- lease-based: выдаёт pending элементы, переводит в processing + устанавливает lease_until
                                      -- Пропускает элементы с idempotency_key в terminal статусе
                                      -- Истёкший lease (processing + lease_until < now) → возврат в pending
POST   /api/sync/ack                  -- подтвердить применение (идемпотентно по idempotency_key)
  Body: {
    "items": [
      {"queue_id": 1, "idempotency_key": "abc", "status": "applied", "bear_id": "xxx"},
      {"queue_id": 2, "idempotency_key": "def", "status": "failed", "error": "Bear not running"}
    ]
  }

POST   /api/sync/attachments/:id      -- загрузить файл вложения
GET    /api/sync/status               -- статус последней синхронизации
```

### 2. Mac-агент (`bear-bridge`, Go)

Запускается на MacBook через launchd каждые 5 минут. Один бинарник, как и хаб — единый стек, общие модели и маппинг.

**Компоненты:**

- **Читатель SQLite** — открывает Bear БД read-only (`?mode=ro`) через `modernc.org/sqlite`:
  `~/Library/Group Containers/9K33E3U3T4.net.shinyfrog.bear/Application Data/database.sqlite`
  Читает: `ZSFNOTE`, `ZSFNOTETAG`, `ZSFNOTEFILE`, `ZSFNOTEBACKLINK`, `Z_5TAGS`, `Z_5PINNEDINTAGS`.

- **Маппинг полей** — конвертирует Core Data имена в чистые имена хаба + формирует `bear_raw` JSON. Ресолвит junction tables (Z_PK → UUID через JOIN). Общий пакет моделей с хабом.

- **Экспортёр дельт** — агент хранит `last_sync_at` локально (файл `~/.bear-bridge-state.json`). Delta-запросы по `ZMODIFICATIONDATE >= last_sync_at` (с overlap-window, `>=` а не `>`, чтобы не терять записи на границе timestamp при округлении или clock drift; дубли безопасны — хаб делает upsert по bear_id). Выполняются **для каждой сущности отдельно:**
  - `ZSFNOTE` — изменённые/новые заметки.
  - `ZSFNOTETAG` — изменённые/новые теги (переименование, pinned и т.д.).
  - `ZSFNOTEFILE` — изменённые/новые вложения.
  - `ZSFNOTEBACKLINK` — изменённые/новые бэклинки.
  - `Z_5TAGS`, `Z_5PINNEDINTAGS` (junction tables) — **не имеют `ZMODIFICATIONDATE`**. Двойная стратегия:
    1. **Delta-цикл (каждые 5 мин):** для каждой изменённой заметки (из delta ZSFNOTE) перечитывать все привязки тегов. Хаб делает DELETE + INSERT привязок **только для заметок, включённых в push** (scope: note_id IN push).
    2. **Full-scan junction tables (каждый 12-й цикл ≈ раз в час):** bridge читает ВСЕ `Z_5TAGS`/`Z_5PINNEDINTAGS`, сравнивает с сохранённым snapshot в state-файле (`known_note_tag_pairs`, `known_pinned_note_tag_pairs`). Для каждой заметки, у которой набор тегов изменился, bridge отправляет в push **полный snapshot тегов этой заметки** (все текущие пары note→tag). Это консистентно с контрактом хаба (DELETE + INSERT по note_id), а не pair-level delta, чтобы не терять связи. Счётчик циклов хранится в `~/.bear-bridge-state.json` (`junction_full_scan_counter`).
  - Для обнаружения удалений — сравнивает множества текущих UUID всех сущностей в Bear с сохранёнными множествами из предыдущего sync (`~/.bear-bridge-state.json` хранит `known_note_ids`, `known_tag_ids`, `known_attachment_ids`, `known_backlink_ids`).
  - Отправляет дельту через `POST /api/sync/push`.

- **Применятель очереди** — забирает `GET /api/sync/queue` (хаб выдаёт pending элементы, переводит в `processing` с lease), для каждого действия:
  - Все вызовы через **xcall** (`brew install xcall`) — CLI утилита, вызывающая x-callback-url и возвращающая x-success результат.
  - Bear token читается из env `BEAR_TOKEN` (Bear 1.7+). Не передаётся через CLI-аргументы bear-bridge. **Внимание:** при вызове xcall token передаётся в URL-аргументе дочернего процесса (inherent exposure, см. Security considerations).
  - `create` -> `xcall -url "bear://x-callback-url/create?token=...&title=...&text=...&tags=..."` → возвращает JSON с `identifier` (UUID новой заметки)
  - `update` -> `xcall -url "bear://x-callback-url/add-text?token=...&id=<bear_id>&text=...&mode=replace"` (bear_id из хаба)
  - `add_tag` -> `xcall -url "bear://x-callback-url/add-tag?token=...&id=<bear_id>&tag=..."`
  - `remove_tag` -> **ОТЛОЖЕНО до Фазы 3.** Bear x-callback-url не имеет эндпоинта для удаления тега. Единственный workaround (body rewrite через `add-text?mode=replace`) рискует повредить inline-теги в тексте заметки. Требуется исследование безопасного подхода (парсинг Bear Markdown формата, учёт nested/inline тегов).
  - **Идемпотентность openclaw→hub:** queue item создаётся с client `Idempotency-Key`. Повторный HTTP retry openclaw не создаёт дублей.
  - **Lease-модель (hub→bridge):** хаб устанавливает `lease_until` при выдаче. Если bridge не прислал ack до expiry — элемент возвращается в `pending` (at-least-once доставка).
  - **Duplicate-safe apply:** перед выполнением bridge проверяет, не достигнуто ли уже целевое состояние в Bear (например, `create` уже создал заметку с данным `bear_id`, `update` уже применён по hash body, `add_tag` уже присутствует). Если состояние уже достигнуто — bridge делает ack `applied` без повторного вызова x-callback.
  - **Верификация create:** UUID получается из x-success ответа xcall. Fallback (если xcall недоступен): ждёт 2 сек, ищет в Bear SQLite по `title + ZCREATIONDATE > now-5sec`. Если >1 результат — `failed`.
  - **Верификация update/add_tag:** ждёт 2 сек, перечитывает Bear SQLite, проверяет что операция применилась по bear_id.
  - После подтверждения — `POST /api/sync/ack` с `idempotency_key`, статусом `applied` + `bear_id` (UUID из Bear). Хаб заполняет `notes.bear_id`. Если не подтвердилось — `failed` с описанием ошибки.

**Зависимости:** xcall — `brew install xcall`. Добавить проверку наличия xcall при старте bridge.

**Расписание:** launchd plist, `StartInterval: 300` (каждые 5 мин).

**Защита от параллельного запуска:** `flock(2)` через `syscall.Flock` на файле `~/.bear-bridge.lock`. При старте:
1. Открыть lock-файл (`os.OpenFile` с `O_CREATE`).
2. Попытаться `syscall.Flock(fd, LOCK_EX|LOCK_NB)`.
3. Успех → продолжить работу. Файл остаётся залоченным на всё время процесса.
4. `EWOULDBLOCK` → другой экземпляр работает → выход.
5. При завершении (нормальном или crash) — ОС автоматически снимает flock. Нет stale locks.

## Потоки данных

### Чтение (Bear → openclaw)
1. Пользователь редактирует заметку в Bear
2. Mac-агент (каждые 5 мин) читает SQLite, видит изменение по `ZMODIFICATIONDATE`
3. Mac-агент маппит поля Bear → Hub, формирует `bear_raw` JSON
4. Mac-агент → `POST /api/sync/push` → хаб обновляет заметку
5. openclaw при следующем запросе видит свежие данные

### Запись (openclaw → Bear) — dual-id lifecycle
1. openclaw → `POST /api/notes` → хаб генерирует `id` (hub UUID), `bear_id=NULL`, `sync_status: pending_to_bear` + запись в `write_queue`
2. Заметка сразу доступна по hub `id` для чтения через API (bear_id ещё не назначен)
3. Mac-агент (когда онлайн) → `GET /api/sync/queue` → получает задачи (lease-based, переход в processing)
4. Mac-агент → xcall `bear://x-callback-url/create?...` → получает Bear UUID из x-success
5. Mac-агент → `POST /api/sync/ack` с `bear_id` → хаб обновляет `notes.bear_id`, `sync_status: synced`

### Запись когда Mac офлайн
1. openclaw → `POST /api/notes` → хаб сохраняет с hub UUID, `bear_id=NULL`, заметка доступна для чтения сразу по hub id
2. `write_queue` копит операции
3. Mac подключается → Mac-агент забирает всю очередь (lease) → применяет пачкой → подтверждает с bear_id → хаб заполняет bear_id

### Initial sync (первый запуск)
1. Агент обнаруживает отсутствие `~/.bear-bridge-state.json` — режим initial sync
2. Читает все заметки из Bear, отправляет батчами по 50 через `POST /api/sync/push`
3. Каждый batch — partial push, хаб создаёт записи с hub UUID + bear_id из Bear
4. После завершения — сохраняет `last_sync_at` и множество bear_id в state-файл
5. В `sync_meta` хаба записывается `initial_sync_complete=true`

## Ограничения и edge cases

**Лимит длины x-callback-url:**
- URL имеет практический лимит длины. Заметки с body >~50 KB после URL-encoding могут не пройти.
- Крупные заметки, которые не удалось обновить через x-callback-url — помечаются как `failed` в write_queue с описанием причины.
- Требуется исследование альтернативных подходов для больших заметок (clipboard, temp-файл) на этапе Фазы 2.

**Зашифрованные заметки:**
- API возвращает `403 Forbidden` при попытке обновить (`PUT /api/notes/:id`) или удалить заметку с `encrypted=1`.
- Зашифрованные заметки — read-only для openclaw (body пустой, данные в `encrypted_data`).
- Создание зашифрованных заметок через openclaw не поддерживается.

## Обработка конфликтов

**Одновременное редактирование (openclaw + пользователь):**
- При `sync/push` Mac-агент отправляет `modified_at` из Bear
- Хаб сравнивает: если заметка изменена и openclaw, и пользователем — помечает `sync_status: conflict`
- Mac-агент НЕ применяет конфликтные изменения
- Создаёт отдельную заметку `"[Conflict] Original Title"` с версией openclaw
- Пользователь разбирает вручную

**Mac долго офлайн:**
- Очередь персистентная (SQLite), ничего не теряется
- При подключении применяется пачкой, ORDER BY created_at
- `GET /api/sync/status` показывает размер очереди и время последнего синка

**Bear переустановлен / база сброшена (reconciliation):**
- **Обнаружение:** при sync bridge сравнивает bear_id из Bear с сохранённым множеством в state. Если >80% bear_id не найдены → режим reconciliation.
- **Matching:** bridge строит маппинг `old_bear_id → new_bear_id` по content hash:
  - Заметки: `SHA256(title + body)` — уникальность контента.
  - Теги: по `title` (уникален в Bear).
  - Вложения: по `filename + file_size + note content hash`.
- **Reconcile endpoint:** `POST /api/sync/reconcile` принимает маппинг `{"notes": {"old_bear_id": "new_bear_id", ...}, "tags": {...}, "attachments": {...}, "backlinks": {...}}`.
- **Hub обновляет только `bear_id`** — PK (hub id) не меняется, FK-ссылки остаются валидными. В одной транзакции обновляются `notes.bear_id`, `tags.bear_id`, `attachments.bear_id`, `backlinks.bear_id`.
- **Backlinks fallback:** если однозначный mapping backlinks недоступен, bridge отправляет `rebuild_backlinks=true`, хаб делает `DELETE FROM backlinks` и ждёт полный re-push backlinks из Bear.
- **write_queue:** элементы ссылаются на hub id (не bear_id), поэтому не затрагиваются при reconciliation.
- **Реализация в Фазе 3** (не MVP). До этого Bear reset требует ручного вмешательства.

**Большие вложения:**
- Хранятся на диске VPS (не BLOB). Mac-агент синкает отдельно через `POST /api/sync/attachments/:id`.
- При `permanently_deleted=1` — периодическая задача в хабе удаляет файлы с диска VPS (при sync push или по расписанию).

## Security considerations

**Bear token в x-callback-url:**
- **Inherent exposure:** xcall принимает URL как CLI-аргумент → token виден в `ps` на время жизни процесса xcall. Это неустранимое ограничение протокола x-callback-url: token передаётся в query string, а xcall не поддерживает альтернативных способов передачи URL (stdin, файл).
- **Разграничение:** `BEAR_TOKEN` читается из env-переменной и не передаётся через CLI-аргументы самого bear-bridge бинарника. Однако при вызове xcall token неизбежно оказывается в argv дочернего процесса.
- **Оценка риска:**
  - Процесс xcall живёт миллисекунды — окно для перехвата через `ps` минимально.
  - Mac-агент запускается под пользователем, `ps` по умолчанию на macOS показывает процессы только текущего пользователя.
  - Bear token имеет ограниченный scope: только Bear app на локальном Mac, бесполезен без физического доступа к машине.
  - Вектор атаки: локальный злоумышленник с доступом к тому же пользователю. При таком уровне доступа атакующий может напрямую читать Bear SQLite — token не даёт дополнительных возможностей.
- **Митигации в коде:**
  - В логах bridge: URL x-callback маскируется — `token=***` вместо реального значения.
  - Уровень логирования для x-callback вызовов — `Debug` (не включен по умолчанию в production).
  - **TODO (Фаза 2):** исследовать альтернативы xcall, которые принимают URL через stdin или файл (например, кастомный Swift CLI через `NSWorkspace.open(url:)`).

**API токены хаба:**
- Хранятся в env-переменных на VPS и Mac.
- Не логируются, не включаются в ответы API.
- Отдельные токены для Mac-агента и openclaw с разными правами.

- Env-переменные для конфигурации (без файлов конфигов)

## Тестирование

### Unit-тесты
- **Фреймворк:** `testing` + `github.com/stretchr/testify` (assert/require)
- **Моки:** `moq` для интерфейсов (Store, BearDB, HubClient, XCallback)
- **Покрытие:** `make test-coverage` (HTML-отчёт)
- **Race detection:** `make test-race`
- **Что тестируем:**
  - `internal/mapper/` — маппинг Bear → Hub, конвертация Core Data дат → ISO 8601, формирование bear_raw JSON
  - `internal/store/` — CRUD операции, FTS5 поиск, write_queue, sync_meta, триггеры
  - `internal/api/` — HTTP handlers через `httptest.NewServer`, авторизация (Bearer token), валидация, пагинация, фильтры

  - `internal/xcallback/` — формирование URL, экранирование параметров, добавление Bear token

### E2E-тесты
- **Полный цикл чтения:** Bridge читает тестовую Bear SQLite → push на Hub → openclaw читает через API → проверка данных
- **Полный цикл записи:** openclaw пишет через API → write_queue → Bridge забирает очередь → ack → проверка статусов
- **Фикстуры:** тестовая Bear SQLite в testdata/ (несколько заметок, теги, вложения, бэклинки, зашифрованная заметка)
- **Hub:** запускается in-process на рандомном порту (`net.Listen("tcp", ":0")`)
- **Проверки:** данные корректно проходят весь pipeline, FTS5 работает, фильтры/пагинация работают, зашифрованные заметки read-only
- **Junction table full-scan:** тест на кейс "связи тегов изменились, но `ZMODIFICATIONDATE` заметки не изменился":
  1. Initial sync → hub имеет заметку A с тегами [X, Y].
  2. В тестовой Bear SQLite: добавить пару (A, Z) в `Z_5TAGS` и удалить (A, Y), **не меняя** `ZMODIFICATIONDATE` заметки A.
  3. Delta-цикл bridge → push НЕ включает заметку A (modified_at не изменился) → hub по-прежнему [X, Y].
  4. Full-scan цикл bridge → обнаруживает diff → push полный snapshot тегов A = [X, Z] → hub обновляет → проверка: hub имеет [X, Z].

### Фаза 2: Запись
- Очередь записи в хабе (write_queue с idempotency_key, effectively-once для openclaw API + at-least-once hub→bridge)
- Prerequisite: `brew install xcall` на Mac
- Mac-агент: применение очереди через xcall + x-callback-url (Bear token из env, верификация через перечитывание SQLite)
- Idempotency API: обязательный `Idempotency-Key` для POST/PUT/DELETE (dedupe на хабе)
- At-least-once delivery hub→bridge + duplicate-safe apply checks в bridge
- Запрет записи зашифрованных заметок (403 Forbidden)
- Исследование лимита x-callback-url на больших заметках
- Security: маскировка token в логах, Debug-level для x-callback URLs
- openclaw: создание/обновление заметок
- Unit-тесты: xcallback, hubclient, idempotency
- E2E-тесты: полный цикл записи; crash-recovery (apply без ack → повторный pickup без дублирования); HTTP retry с тем же Idempotency-Key не создаёт дубль

### Фаза 3: Вложения + конфликты + reconciliation + remove_tag
- Синхронизация файлов вложений на VPS
- Очистка файлов при permanently_deleted
- Обнаружение и разрешение конфликтов
- Bear reset reconciliation: обнаружение reset, content-hash matching, `POST /api/sync/reconcile`, UPDATE bear_id для notes/tags/attachments/backlinks (PK не меняется), fallback rebuild backlinks
- `DELETE /api/notes/:id/tags/:tag_id` — реализация remove_tag с безопасным парсингом Bear Markdown
- Исследование альтернатив xcall (Swift CLI через NSWorkspace) для устранения token exposure в argv
- Мониторинг и статусы синхронизации

## Решения и обоснования

| Решение | Почему |
|---|---|
| Go для хаба | Один бинарник, минимум RAM (~10 MB), простой деплой на VPS |
| Go для Mac-агента | Единый стек с хабом, общие модели/маппинг, один бинарник без зависимостей |
| SQLite для хаба | Достаточно для 2000 заметок, FTS5, без внешних зависимостей |
| Hub-модель (не direct sync) | Decoupling: Mac-агент и openclaw независимы, легко менять компоненты |
| Вложения на диске, не BLOB | Эффективнее для больших файлов, проще бэкапить |
| Чистая схема + raw JSON (Вариант C) | Читаемый API + полная целостность данных (bear_raw как страховка) |
| Dual-id (hub UUID PK + bear_id UNIQUE) | Hub UUID стабилен как PK (не меняется при Bear reset). bear_id заполняется при sync. Заметки от openclaw доступны до sync с Bear |
| ON DELETE CASCADE на FK | Безопасное каскадное удаление junction-таблиц при удалении notes/tags. Не нужен ручной порядок DELETE |
| Все поля Bear включая encrypted | Полная копия данных, даже если openclaw не может расшифровать |
| Backlinks включены | Полная структура графа заметок для навигации и RAG |
